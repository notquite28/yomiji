# UI Fix Plan: Review Keyboard Avoidance + Quick Settings Alignment

Created: 2026-05-18 13:36:16

## Scope

Plan fixes for two UI issues without implementing code yet:

1. Long vocabulary prompts can cause the on-screen keyboard to hide the answer textbox.
2. The Review Quick Settings modal header is visually misaligned, especially the centered title and `Done` button.

Relevant files inspected:

- `src/screens/ReviewSessionScreen.tsx`
- `src/components/ScreenLayout.tsx`
- `src/components/ReviewQuickSettings.tsx`
- `src/components/SubjectHeroCard.tsx`
- `app.json`

Note: both provided clipboard image paths rendered as the same keyboard screenshot in the agent environment, so the modal issue is planned from code inspection and the user's description.

## Issue 1: Keyboard hides answer textbox for large vocabulary cards

### Observed / likely behavior

In review sessions, a tall vocabulary hero card can push the answer `TextInput` low enough that the Android soft keyboard overlays it. The visible screenshot shows the keyboard covering the lower portion of the review screen, with the answer area partially/fully hidden behind it.

### Relevant current code

`ReviewSessionScreen` renders the session inside:

```tsx
<ScreenLayout scrollable keyboardShouldPersistTaps>
  <SessionHeader ... />
  <SubjectHeroCard ... />
  <TextInput ... />
  ...
</ScreenLayout>
```

`ScreenLayout` currently handles scrollable mode with a `SafeAreaView` and `ScrollView` only:

```tsx
<SafeAreaView style={styles.safeArea}>
  <ScrollView
    contentContainerStyle={styles.scrollContent}
    keyboardShouldPersistTaps={keyboardShouldPersistTaps ? 'handled' : undefined}
  >
    {children}
  </ScrollView>
</SafeAreaView>
```

There is no `KeyboardAvoidingView`, explicit keyboard inset padding, or Android keyboard resize configuration in `app.json`.

### Likely root cause

The screen is scrollable, but when the keyboard opens React Native is not guaranteed to resize or pad the visible area enough for the focused `TextInput`. On Android this is especially dependent on window soft-input mode. Because the hero card has a fixed default `minHeight = 240` and large Japanese text, the answer field can be placed near the keyboard boundary on smaller screens or with large keyboards.

### Proposed fix

#### A. Add keyboard-aware layout support to `ScreenLayout`

Enhance `src/components/ScreenLayout.tsx` so scrollable screens can avoid the keyboard.

Implementation direction:

- Import `KeyboardAvoidingView` and `Platform` from `react-native`.
- In `scrollable` mode, wrap the `ScrollView` with `KeyboardAvoidingView`.
- Use `behavior="padding"` on iOS.
- For Android, either omit behavior or use a conservative setup that relies on resize mode plus extra scroll padding.
- Add a bottom padding value to the scroll content so the focused input can scroll above the keyboard.

Potential structure:

```tsx
<SafeAreaView style={styles.safeArea}>
  <KeyboardAvoidingView
    style={styles.keyboardAvoiding}
    behavior={Platform.OS === 'ios' ? 'padding' : undefined}
  >
    <ScrollView
      contentContainerStyle={styles.scrollContent}
      keyboardShouldPersistTaps={keyboardShouldPersistTaps ? 'handled' : undefined}
    >
      {children}
    </ScrollView>
  </KeyboardAvoidingView>
</SafeAreaView>
```

Add style:

```tsx
keyboardAvoiding: {
  flex: 1,
}
```

And consider increasing `scrollContent` bottom padding:

```tsx
scrollContent: {
  flexGrow: 1,
  padding: 20,
  paddingBottom: 40,
  gap: 18,
}
```

If this affects non-keyboard scrollable screens too much, add an explicit prop such as `keyboardAvoiding?: boolean` and enable it only for review/lesson screens.

#### B. Set Android keyboard resize mode in Expo config

Consider adding this under `expo.android` in `app.json`:

```json
"softwareKeyboardLayoutMode": "resize"
```

Rationale:

- Expo supports `softwareKeyboardLayoutMode` for Android.
- `resize` makes the app window resize when the keyboard appears instead of letting the keyboard float over content.
- This is usually the desired behavior for forms and study answer inputs.

Caveat:

- This is an app-level behavior change. Verify login/settings text inputs still behave correctly.

#### C. Optional review-specific polish

If keyboard avoidance alone is not enough on compact screens, reduce the hero card height only while answering:

```tsx
<SubjectHeroCard minHeight={220} ... />
```

Or compute a more compact value for vocabulary/review mode. This should be a fallback, not the first fix, because the primary bug is keyboard/layout behavior rather than hero size.

### Validation checklist

Manual QA:

- Android physical/emulator device.
- Start a review session with a long vocabulary prompt.
- Focus the answer input.
- Confirm the textbox remains visible above the keyboard.
- Confirm user can scroll the input and submit button into view.


## Issue 2: Quick Settings modal title / Done button alignment

### Observed / likely behavior

The quick settings modal header appears visually misaligned. The title and `Done` button do not share a balanced header layout.

### Relevant current code

`src/components/ReviewQuickSettings.tsx` renders a modal:

```tsx
<Modal
  visible={visible}
  animationType="slide"
  presentationStyle="pageSheet"
  onRequestClose={onClose}
>
  <View style={styles.container}>
    <View style={styles.header}>
      <Text style={styles.title}>Quick Settings</Text>
      <Pressable onPress={onClose} style={styles.closeButton} ...>
        <Text style={styles.closeButtonText}>Done</Text>
      </Pressable>
    </View>
    ...
  </View>
</Modal>
```

Current header styles:

```tsx
container: {
  flex: 1,
  backgroundColor: theme.colors.background,
  paddingTop: 20,
},
header: {
  flexDirection: 'row',
  alignItems: 'center',
  justifyContent: 'space-between',
  paddingHorizontal: 20,
  paddingBottom: 16,
},
title: {
  color: theme.colors.text,
  fontSize: 22,
  fontWeight: '900',
},
closeButton: {
  borderRadius: 999,
  paddingHorizontal: 16,
  paddingVertical: 8,
  backgroundColor: theme.colors.surface,
  borderWidth: 1,
  borderColor: theme.colors.border,
},
```

### Likely root cause

- The modal content is not wrapped in `SafeAreaView`, so top spacing relies on hardcoded `paddingTop: 20`.
- The header uses `justifyContent: 'space-between'`, which places the title on the left and `Done` on the right. If the desired design is a centered modal title with a trailing action, the title will appear off-center.
- The `Done` button has no fixed/min height or width, so vertical and horizontal optical balance may vary.

### Proposed fix

#### A. Wrap modal content in `SafeAreaView`

Import:

```tsx
import { SafeAreaView } from 'react-native-safe-area-context';
```

Then change modal body to:

```tsx
<SafeAreaView style={styles.container} edges={['top', 'bottom']}>
  ...
</SafeAreaView>
```

Remove the hardcoded `paddingTop: 20` from `container` or reduce it to a small intentional spacing if needed.

#### B. Rebuild header as a three-column layout

Use a stable left spacer, centered title, and right action.

Potential JSX:

```tsx
<View style={styles.header}>
  <View style={styles.headerSide} />
  <Text style={styles.title}>Quick Settings</Text>
  <View style={styles.headerSide}>
    <Pressable onPress={onClose} style={styles.closeButton} ...>
      <Text style={styles.closeButtonText}>Done</Text>
    </Pressable>
  </View>
</View>
```

Potential styles:

```tsx
header: {
  minHeight: 56,
  flexDirection: 'row',
  alignItems: 'center',
  paddingHorizontal: 20,
  paddingBottom: 12,
},
headerSide: {
  width: 88,
  alignItems: 'flex-end',
},
title: {
  flex: 1,
  color: theme.colors.text,
  fontSize: 20,
  fontWeight: '900',
  textAlign: 'center',
},
closeButton: {
  minHeight: 38,
  borderRadius: 999,
  paddingHorizontal: 16,
  alignItems: 'center',
  justifyContent: 'center',
  backgroundColor: theme.colors.surface,
  borderWidth: 1,
  borderColor: theme.colors.border,
},
```

This centers the title in the available header regardless of the `Done` button width.

#### C. Preserve accessibility

Keep existing accessibility values:

```tsx
accessibilityRole="button"
accessibilityLabel="Close quick settings"
```

No behavioral changes are needed.

## Implementation order if approved

1. Update `ReviewQuickSettings` safe-area/header layout.
2. Update `ScreenLayout` keyboard avoidance.

## Risks / tradeoffs

- Adding app-level Android keyboard resize may subtly affect existing screens with text inputs. This is likely desirable, but should be smoke tested.
- Adding keyboard avoidance to all scrollable `ScreenLayout` users could affect spacing on non-form screens. If this is a concern, make it opt-in and enable for review/lesson sessions only.
- Centering modal title with fixed side widths assumes the `Done` button stays short. If localized text gets longer later, use dynamic measurement or a wider side column.
