## Findings

### ‚úÖ ReviewQuickSettings Header Alignment

SafeAreaView wrapping, three-column header (left spacer / centered title / right Done), vertical
balance via minHeight values ‚Äî all correct. Title genuinely centers regardless of Done button
width. Hardcoded paddingTop removed. Accessibility labels preserved.

### ‚úÖ KeyboardAvoidingView Wiring

The `keyboardAvoiding` prop gates the `KeyboardAvoidingView` wrapper correctly:
- iOS uses `behavior="padding"`
- Android relies on `softwareKeyboardLayoutMode: "resize"` (configured in `app.json`) and gets
  no behavior override
- Non-keyboard screens (`ReviewSummary`, `CenteredMessage`, etc.) are unaffected because they
  don't opt in

### ‚úÖ Android softwareKeyboardLayoutMode

`"softwareKeyboardLayoutMode": "resize"` added to `expo.android`. This makes the app window
resize when the keyboard opens rather than letting the keyboard float over content. Correct
default for form/input-heavy apps.

### ‚úÖ ReviewSessionScreen opt-in

`keyboardAvoiding` prop passed to `<ScreenLayout>` ‚Äî only screen that needs it.

### [P2 ‚Äî RESOLVED] Unconditional paddingBottom: 40 on all scrollable screens

**Original finding:** `scrollContent` had `paddingBottom: 40` applied unconditionally, affecting
every scrollable `ScreenLayout` screen (ReviewSummary, LessonSessionScreen, LessonPickerScreen,
RadicalImagePreviewScreen) ‚Äî all read-only or completion screens with no text input.

**Resolution:** Removed `paddingBottom: 40` from the base `scrollContent` style. The ScrollView's
native scroll-to-focused-input behavior, combined with `KeyboardAvoidingView` (iOS) and
`softwareKeyboardLayoutMode: "resize"` (Android), handles keyboard avoidance without needing
extra bottom padding. Non-keyboard scrollable screens retain their original `padding: 20` only.

If real-world testing on compact Android devices shows the review input still gets obscured,
paddingBottom can be reintroduced conditionally (only when `keyboardAvoiding` is true) via a
custom `contentContainerStyle` override.
